# 下游接口调用接口规范

## 1. 文档说明

### 1.1 文档目的

本文档用于规范下游系统调用主数据平台开放接口的接入方式、认证方式、请求格式、返回格式、错误处理和联调要求。  
当前主数据平台对下游系统开放以下三类只读数据接口：

1. 组织部门主数据接口
2. 员工基本信息主数据接口
3. 员工工作信息主数据接口

### 1.2 适用范围

适用于所有需要从主数据平台拉取基础组织、人事主数据的下游业务系统、集成平台、数据中台、报表平台和流程平台。

### 1.3 设计原则

1. 统一认证：复用平台现有 Bearer Token 认证机制。
2. 统一协议：全部采用 `HTTP + JSON` 交互。
3. 统一风格：全部采用 `POST` 方式提供拉取接口。
4. 统一返回：全部采用统一响应包结构。
5. 统一分页：全部采用页码分页，避免一次性拉取超大结果集。
6. 支持增量：支持基于更新时间水位 `updatedAfter` 的增量拉取。
7. 只读开放：下游仅允许拉取，不允许通过本接口反向写入主数据平台。

### 1.4 版本信息

| 项目 | 内容 |
| --- | --- |
| 文档名称 | 下游接口调用接口规范 |
| 接口版本 | v1 |
| 协议版本 | HTTP/1.1 |
| 数据格式 | `application/json` |
| 字符编码 | `UTF-8` |
| 时间格式 | `yyyy-MM-dd HH:mm:ss` |

## 2. 接口接入总览

### 2.1 服务地址

主数据系统默认服务端口为 `8089`，系统上下文路径为 `/`。  
如未经过网关转发，则下游调用地址格式如下：

```text
http://{主数据服务IP}:8089
```

生产环境建议通过统一域名或网关地址访问，例如：

```text
https://{api-domain}
```

### 2.2 接口清单

| 数据类型 | 接口名称 | 请求方式 | 请求 URL |
| --- | --- | --- | --- |
| 组织部门 | 拉取组织部门主数据 | `POST` | `/mainData/downstream/organizationDepartment/pull` |
| 员工基本信息 | 拉取员工基本信息主数据 | `POST` | `/mainData/downstream/personBasicInfo/pull` |
| 员工工作信息 | 拉取员工工作信息主数据 | `POST` | `/mainData/downstream/personJobInfo/pull` |

### 2.3 完整请求 URL 示例

若主数据系统部署地址为 `10.10.10.20:8089`，则完整示例如下：

```text
http://10.10.10.20:8089/mainData/downstream/organizationDepartment/pull
http://10.10.10.20:8089/mainData/downstream/personBasicInfo/pull
http://10.10.10.20:8089/mainData/downstream/personJobInfo/pull
```

## 3. 认证与安全规范

### 3.1 认证方式

下游系统调用开放接口前，应先通过平台登录接口获取访问令牌，再在业务接口请求头中携带：

```http
Authorization: Bearer {token}
```

### 3.2 Token 获取方式

请求 URL：

```text
POST /login
```

请求头：

```http
Content-Type: application/json
```

请求示例：

```json
{
  "username": "downstream_user",
  "password": "******"
}
```

响应示例：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiJ9..."
  }
}
```

### 3.3 安全要求

1. 生产环境应优先使用 `HTTPS` 传输。
2. Token 应由下游系统安全保管，禁止明文落日志、落前端页面、落共享文档。
3. 建议下游系统使用专门的接口账号，不建议复用人工登录账号。
4. 建议接口调用链路透传请求追踪号，例如：

```http
X-Request-Id: 202603220001
```

## 4. 公共请求规范

### 4.1 请求头规范

所有下游拉取接口统一使用以下请求头：

```http
Content-Type: application/json
Accept: application/json
Authorization: Bearer {token}
```

### 4.2 公共请求参数

三个下游拉取接口统一支持以下公共请求参数：

| 参数名 | 类型 | 是否必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `pageNum` | `Integer` | 否 | `1` | 页码，从 `1` 开始。 |
| `pageSize` | `Integer` | 否 | `200` | 每页条数，最大 `1000`。 |
| `includeInvalid` | `Boolean` | 否 | `false` | 是否包含失效/停用数据。 |
| `updatedAfter` | `String` | 否 | 无 | 增量拉取时间水位，仅返回该时间及之后更新的数据。格式：`yyyy-MM-dd HH:mm:ss`。 |

### 4.3 公共拉取策略

1. 首次拉取建议采用全量方式：`updatedAfter` 不传。
2. 后续拉取建议采用增量方式：传入上次成功拉取的最大更新时间。
3. 下游系统应以“最后一次成功处理完成的时间点”作为下次增量起点。
4. 为避免边界数据遗漏，建议下游增量拉取时做 1 到 5 分钟时间回拨，并在下游侧按业务主键去重。

### 4.4 幂等性说明

虽然接口采用 `POST` 请求，但三类开放接口均为只读拉取接口，不会修改主数据平台数据，业务语义上可视为幂等读接口。

## 5. 公共返回规范

### 5.1 统一响应结构

所有下游接口统一返回两层响应结构：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {
    "total": 2,
    "rows": [],
    "code": 200,
    "msg": "查询成功"
  }
}
```

### 5.2 返回字段说明

#### 第一层响应字段

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `code` | `Integer` | 接口执行结果码，`200` 表示成功。 |
| `msg` | `String` | 接口执行结果说明。 |
| `data` | `Object` | 分页结果对象。 |

#### 第二层分页字段

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `total` | `Long` | 满足条件的总记录数。 |
| `rows` | `Array` | 当前页数据列表。 |
| `code` | `Integer` | 列表查询状态码，正常为 `200`。 |
| `msg` | `String` | 列表查询状态说明。 |

### 5.3 成功判定规则

建议下游系统按以下规则判定接口成功：

1. 最外层 `code = 200`
2. `data.code = 200`
3. `rows` 字段存在且为数组

## 9. 错误码规范

| 错误码 | 含义 | 说明 |
| --- | --- | --- |
| `200` | 成功 | 请求处理成功。 |
| `401` | 未登录或 Token 无效 | 未携带 Token、Token 过期或认证失败。 |
| `500` | 系统内部错误 | 服务端处理异常，请联系主数据系统管理员。 |

错误响应示例：

```json
{
  "code": 401,
  "msg": "认证失败，无法访问系统资源",
  "data": null
}
```

## 10. 下游系统对接建议

### 10.1 推荐拉取模式

1. 首次上线：执行全量拉取。
2. 日常运行：按固定频率执行增量拉取。
3. 异常补偿：按业务主键去重后重新补拉指定时间范围数据。

### 10.2 推荐分页策略

1. 单次建议 `pageSize = 200` 到 `500`。
2. 若数据量大，可采用 `pageSize = 1000` 的上限值。
3. 下游应循环读取直到当前页返回 `rows` 为空或累计条数达到 `total`。

### 10.3 推荐增量策略

1. 使用 `updatedAfter` 作为增量水位。
2. 水位字段建议存储为接口返回结果中的最大 `updateTime`，若 `updateTime` 为空则回退到 `createTime`。
3. 为避免边界遗漏，建议按“上次成功时间减 1 到 5 分钟”重新拉取，并在下游按主键做幂等覆盖。

### 10.4 推荐主键使用方式

1. 组织部门以 `pkDept` 为唯一业务主键。
2. 员工基本信息以 `pkPsndoc` 为唯一业务主键。
3. 员工工作信息以 `pkPsnjob` 为唯一业务主键。

## 12. 联调确认项

建议双方联调时逐项确认以下内容：

1. 登录接口是否可正常获取 Token。
2. Bearer Token 是否可成功访问三个开放接口。
3. 分页参数 `pageNum`、`pageSize` 是否符合预期。
4. 增量参数 `updatedAfter` 是否按 `yyyy-MM-dd HH:mm:ss` 正常解析。
5. `includeInvalid` 为 `false` 时是否正确过滤停用/失效数据。
6. 下游系统是否已按业务主键完成去重与覆盖写入。

## 13. 结论

主数据系统当前已按照统一的开放接口模式，为下游系统提供三类只读拉取接口。  
下游系统可基于统一 Token、统一 URL 风格、统一请求体、统一响应包和统一分页机制完成标准化集成接入。
